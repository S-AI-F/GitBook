3.1 Getting a Git repositery

There are two ways for obtaining a Git repository: - Transform an existing local directory without version control into a Git repositor - Clone an existing Git repositery from somewhere else

3.1.1 Initializig a Repository in an Existing Directory

First, you have to go to the project directory

# work directory
initial_repo = "D:/git_test/Git_Book/"
setwd(initial_repo)
# Repository directory name
dir_name = "MyRepo"
new_repo = paste(initial_repo,"MyRepo", sep = "/")
# verify if the repo file exist and remove it if it exists
if(dir.exists("D:/git_test/Git_Book/MyRepo") == TRUE){
  unlink("D:/git_test/Git_Book/MyRepo", recursive = TRUE)
}
# we create a project directory
dir.create("MyRepo")
## Warning in dir.create("MyRepo"): 'MyRepo' existe déjà
# Then we go to that projetc directory
setwd(new_repo)
shell("dir")
# initialize a git repository
shell("git init")

This creates a new subdirectory named .git that contains all necessary repository files (a git repository skeleton).

Now we can start version-conrolling files and do initial commits.

setwd(new_repo)
# First we have to specify our identity
shell('git config --global user.name "Saif Shabou"')
shell('git config --global user.email "saif.shabou@gmail.com"')
print(getwd())
## [1] "D:/git_test/Git_Book/MyRepo"
# then we create a new file and commit this action
nb = sample(1:1000, 1, replace=TRUE)
filename = paste("file_",nb,".csv", sep = "")
file.create(filename)
## [1] TRUE
cmd_add = paste("git add", filename, sep = " ")
shell(cmd_add)
shell('git commit -m "adding new file"', intern = TRUE)
## [1] "[master (root-commit) 5d83e06] adding new file"  
## [2] " 1 file changed, 0 insertions(+), 0 deletions(-)"
## [3] " create mode 100644 file_933.csv"

3.2 Recording Changes to the repositery

Each file in our working directory can be in one of to possible states: tracked or untracked. Tracked files are the files known by Git. They can be: unmodified, modified, or staged. Untracked files are any files in the orking directory that are not in the staging area. When we clone a repository, all the files will be tracked and unmodified. When we are editing files, Git considers them as modified since they changed from the last commit. As we work, we stage these modified files and then we commit all those stages changes. And this cycle repeats.

Git lifecycle (source:https://git-scm.com/book/en/v2)

3.2.1 Checking the Status of Files

The command git status enables s to determine the status of the different tracked files.

shell("git status", intern = TRUE)

## [10] "\tmodified:   _book (new commits)"

We have to check the file status to know if we need to make a new commit. When the message tells:Your branch is up-to-date with 'origin/master'. nothing to commit, working directory clean, it means that none of our tracked files are modified.

3.2.2 Tracking new Files

In order to track new file, we can use the command git add. To track all the existing files in the directory we use git add . In the git status results, we can see that Git recognizes the changes to be committed

shell("git add .", intern = TRUE)
## character(0)

We can use the command git diff to know what files we changed but not yet staged and what files we staged but not yet commited.

shell("git diff", intern = TRUE)
## character(0)

The command compares what is in our working directory with the staging area. The results show the changes we have made and have not yet staged.

3.2.3 Committing changes

Committing changes include only files created and modified that are added using git add command. We have to check git status before committing in order to check that files to commit are already staged.

shell('git commit -m "my message of commit"', intern = TRUE)
## [1] "[master a05eb2e] my message of commit"              
## [2] " 6 files changed, 151 insertions(+), 4 deletions(-)"

3.4 Working with remotes

Remote repositories are versions of our project that are hosted online.

3.4.1 Showing remotes

We can use the command git remote in order to show the URL of remote servers configured with our project.

shell("git remote -v", intern = TRUE)
## [1] "origin\thttps://github.com/S-AI-F/GitBook.git (fetch)"
## [2] "origin\thttps://github.com/S-AI-F/GitBook.git (push)" 
## [3] "ss\thttps://github.com/S-AI-F/GitBook.git (fetch)"    
## [4] "ss\thttps://github.com/S-AI-F/GitBook.git (push)"

3.4.2 Adding remote repositories

We can add a remote Git repository and reference it with a shortname (here we will use ss)

shell("git remote add ss https://github.com/S-AI-F/GitBook.git")
## Warning in shell("git remote add ss https://github.com/S-AI-F/GitBook.git"):
## 'git remote add ss https://github.com/S-AI-F/GitBook.git' execution failed with
## error code 128
shell("git remote -v", intern = TRUE)
## [1] "origin\thttps://github.com/S-AI-F/GitBook.git (fetch)"
## [2] "origin\thttps://github.com/S-AI-F/GitBook.git (push)" 
## [3] "ss\thttps://github.com/S-AI-F/GitBook.git (fetch)"    
## [4] "ss\thttps://github.com/S-AI-F/GitBook.git (push)"

Now we can use the ss shortname instead of the whole url. We can for example fetch the changes that are not yet in our repository.

shell("git fetch ss", intern = TRUE)
## character(0)

3.4.3 Fetching and pulling from remotes

The command fetch fetches new work that has been pushed to the server and are not merged with our local repository. It only downloads the data to our local repository without merging it with our work and without modifying our work directory. We have to merge it manually after the fetch.

The git pull makes both: fetching and merging changes in the remote branch with our current branch.

3.4.4 Pushing to remotes

We push our work when we think that our project is at a point that we want to share it. For tat, we use the command git push <remote> <branch>. Cloning a remote project sets up generaly the remote name as “origin” and the local branch name as “master”. So if we want to push our master branch to our origin server, we runn the command below to ush any commits we made to the server:

shell("git push origin master", intern = TRUE)
## Warning in system(cmd, intern = intern, wait = wait | intern,
## show.output.on.console = wait, : l'exécution de la commande 'C:
## \WINDOWS\system32\cmd.exe /c git push origin master' renvoie un statut 1
## character(0)
## attr(,"status")
## [1] 1

If another person working on the remote repository pushed information before between our clone and our push, we have to integrate (fetch) their work first before pushing.

3.4.5 Inspecting a remote

We can use the command git remote show <our remote> in order to show information about our remote

shell("git remote show origin", intern = TRUE)
## [1] "* remote origin"                                         
## [2] "  Fetch URL: https://github.com/S-AI-F/GitBook.git"      
## [3] "  Push  URL: https://github.com/S-AI-F/GitBook.git"      
## [4] "  HEAD branch: master"                                   
## [5] "  Remote branch:"                                        
## [6] "    master new (next fetch will store in remotes/origin)"
## [7] "  Local ref configured for 'git push':"                  
## [8] "    master pushes to master (local out of date)"

The command shows the URL of remote repository and tells us which branch is automatically pushed to when we run git push and which remote branches on the server we don’t have yet.

3.4.6 Renaming and removing remotes

We can use the command rename to rename a remote. For example, we will change ss to saif

shell("git remote", intern = TRUE)
## [1] "origin" "ss"
shell("git remote rename ss saif", intern = TRUE)
## character(0)
shell("git remote", intern = TRUE)
## [1] "origin" "saif"

We can use the remove command to remove a remote

shell("git remote remove saif", intern = TRUE)
## character(0)
shell("git remote", intern = TRUE)
## [1] "origin"
